Introduction¶

The objective of this project is to analyze this dataset, an aggregation of responses to medical questions containing roughly 214,000 rows and 39 columns, centering around the effect of Alzheimers on patients ages 50+. I chose this dataset with the goal of extracting trends regarding an illness I have no prior knowledge of, besides what Alzheimers is and how it worsens memory function for the older side of the population. The idea is that having little previous knowledge of these trends from any other sources, I won't know "where to look", and will be a relatively unbiased analyser.

Aslo of note is that I have basically no information about the questions asked or values contained in the dataset, besides the names of the column headers. I had a lot of trouble finding an actual description of column values or the questions/codes within. The data was sourced from the CDC, specifically the Behavioral Risk Factor Surveillance System (BRFSS), which I've found quite tricky to navigate. The best I dug up was this code book, which matches to a few of the codes discovered below (such as C01 for the Overall Health class), but not much else. The only other piece of information I have is the dataset title: "Alzheimer's Disease and Healthy Aging Data" from 2015-2020.

I've decided to roll with it! I believe 'just figuring it out as you go' is a severely underrrated skill, and one most of us will be forced to use many times during our CS career. It isn't a skill that can be tought from theory, like regression, but only by experience; this project is designed as a guide to 'just figuring out' an enormous dataset with little context beforehand, and seeing what we can do with it. It makes the objective two-fold: figure out what the data is representing in the first place/how it's organized, and seeing if we can't find any trends within and make some observations.

Our first step is to import the dataset (I do this via a local csv file which is quicker but it works just as fine with the web link; just uncomment the 'path =' code below). Let's get started with identifying important columns.

Figuring out duplicates and codes¶

We've dealt with the trivial stuff. Let's deal with duplicate columns (and make sure we're not throwing away useful data). While the codebook will, ironically, not actually help us decipher the codes, we easily see a text-code pairing for the following columns: Class, Topic, Question, Location (state), DataValueType, and stratification. Let's see if these have anything to do with each other. Let's get a few functions to help us out.

By far the most important pandas function we use will by .groupby(). It allows easy management of data with many attributes (which our's certainly has); the first step to solving a rubix cube is being able to twist individual faces of our choice instead of punching it until something moves. A full explanation of groupby and it's many uses as well as proper syntax can be found here, but on a high level applying groupby on a datafram (df.groupby(column)) will split the data into a bunch of mini dataframe, one for each value of column. Groupby tends to follow the "split-apply-combine" formulation, where we split into mini dataframes based on the column values, apply some operation to the individual dataframes, and combine them back into one. We have our first use below.

  • group will take two attributes and tell us how many of the second attribute there are for each first attribute. Useful for visualizing the distribution of attributes, like what kind of Topics there are for each Class
  • comp_vars works off of group, first showing us the raw value counts for each attribute, then performing the distribution analysis. It's useful for checking the relationship between two columns, in the case of a State Name and State Abbreviation, for example
In [277]:
#For each grpby value in data, the number of cmpby values is listed. Basically divides data by grpby and divides those further by cmpby
def group(data, grpby, cmpby):
    #split by grpby value, take the cmpby values for each dataframe, and perform value count operations on each one.
    #Here we skip the combine step, just printing out useful info
    by_grp = data.groupby(grpby)[cmpby]
    info = pd.DataFrame(by_grp.value_counts())
    return info

#Shows the raw value counts for each variable within the entire dataset, then calls group (showing the distribution between the two variables)
def comp_vars(data, var1, var2):
    print("Value counts for " + BOLD + var1 + END)
    print("==============================================================================")
    print(data[var1].value_counts())
    print("==============================================================================")
    print("\nValue counts for " + BOLD + var2 + END)
    print("==============================================================================")
    print(data[var2].value_counts())
    print("==============================================================================")
    print("\nComparing " + BOLD + var1 + END + " and " + BOLD + var2 + END)
    print("==============================================================================")
    print(group(data, var1, var2))
    print("==============================================================================")

Using our function, dividing the data by the Class value and then further by the ClassID value, we find a one to one relationship between all text-code pairs. There is one ClassID for each Class, and vice versa. This is good news because it means both columns contain the exact same information, as we'd expect.

(e.g. the class 'Mental Health' has the one code 'C05' and vice versa. If a class had multiple codes, we would see multiple codes appear under 'ClassID' for that class. If a code corresponded to multiple classes, would see duplicates in the 'ClassID' column. This was checked for all other text-code columns, rest assured)

We perform this analysis for many column pairs, and decide to throw away the code columns for variables that have short names like State, Class, Topic, etc (as one column is sufficient to provide us all information) but keep the code for the Questions, which are annoying to work with as strings.

That's a lot of missing values. But, not actually a great measure of the data we have access to. As you may have noticed, our data is given somewhat raw for each state and year, and further stratified by Age Group, and Sex or Ethnicity. At the base level, for a specific Year, State, Sex, etc. we have the raw data, but much is combined or aggregated into Overall rows.

Let's get rid of the State column. Normally we'd have to group the data and sum over all states, but lucky for us BRFSS aslso gives aggregated columns. This is our saving grace; that the people over at the CDC took some of the "dealing with empty values" off our hands. They aggreagate by state, age, etc, presumably performing reasonable imputations; if there was not enough data taken on white males for question QXX in the 50-64 age group in Maryland during 2017, well, there are many ways of inferring the missing info with reasonable guesses so the entire dataset does not become useless. You could average over the rest of the males or ethnicities for that question, use last year's data, or use the average of the rest of the data as a reasonable guess. If you don't have enough data for the 50-64 year olds or 65+ year olds individually, you still might have enough for the overall age group.

Another thing we don't have that we would need to aggregate the data is the sample sizes: if 20/50 people (40%) surveyed in New York said they didn't like pizza (an outrageous ratio) and 4/5 (80%) people in Maryland answered the same, we would not have 60% not liking pizza, it would be closer to 40% since more New Yorkers were surveyed. We're only given data in percents or means, and would have to perform a weighted average with sample size to correctly aggregate (there was a sample size column that came with the dataset but it was unfortunately barren).

We'll do the same thing for ethnicity, which we're not that interested in right now. Similarly, we'll get rid of the rows for which "StratValue2" is an ethnicity. We'll keep the divide on male vs female but make it it's own column. At this point we also know StratCat1 is just age group so we'll get around to removing that label.

Attempting to pick out trends within the Cognitive Decline class and gain some insight on the effects of Alzheimer's, we'll make the data visual

Looks promising, but there's a better way to do this. The above shows why bar plots are very meh. You can compare within the plot, but each plot uses different y-values. We could set the range of each one manually, but it's still tough to look at. There's just a lot of wasted space.

We'll use a line plot instead. In addition, we'll make it plot any question over time, and we'll also have the option to split the data by Age.

  • plot_question_over_time This function can use the data array, question ID, and split that you give it, and perform many of the above calculations automatically while making everything look all pretty. Namely, it identifies the correct question entires, gets rid of the extra information from whichever category we did not give it (either Age or Sex), groups the data by the category we did give it, and plots the values over time for each group

def plot_question_over_time(data, question, group):
    print("\n====================================================================================")
    #We don't want Overall to show up as a year
    data = data[data["TimePeriod"] != "Overall"]
    q = data[(data["QID"] == question)].copy(deep = True)
    q.sort_values(by = ["TimePeriod", "Age", "Sex"], inplace = True)
    
    #Get rid of extra info (only use the aggregated rows) for the other category
    groups = ["Age", "Sex"]
    groups.remove(group)
    rm = groups[0]
    q = q[q[rm] == "Overall"]
    
    #Print out the actual question for context
    print(BOLD + question + ": " + END + key[question])
    #Group by the given category and plot the question value over time for each group
    fig, ax = plt.subplots(figsize = (8, 4))
    grouped = q.groupby(group)
    for g, frame in grouped:
        ax.plot(frame["TimePeriod"], frame["Value"], label = g)
    plt.legend(bbox_to_anchor = (1, 1))
    ax.set_xlabel("Year")
    ax.set_title("Responses to Q31 split by %s over time" % group)
    plt.show()
    print("====================================================================================")
